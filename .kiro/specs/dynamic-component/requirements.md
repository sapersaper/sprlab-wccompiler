# Requirements Document

## Introduction

This feature adds a `<component :is="expr">` directive to the WCC template syntax, enabling dynamic component rendering based on a reactive expression. When the expression value changes, the old element is destroyed and a new one is created via `document.createElement(newTag)`. This eliminates the need for `if`/`else-if` chains when implementing routing or any pattern where the rendered component depends on runtime state.

## Glossary

- **Compiler**: The wcCompiler pipeline that transforms `.wcc` single-file components into native web component JavaScript files
- **Dynamic_Component**: A `<component :is="expr">` element in the template that renders a custom element whose tag name is determined at runtime by evaluating the `:is` expression
- **Tree_Walker**: The compiler subsystem (`tree-walker.js`) that walks the parsed DOM tree to discover bindings, events, and directives
- **Codegen**: The compiler subsystem (`codegen.js`) that generates the final JavaScript output from parsed bindings and metadata
- **Template_Normalizer**: The compiler subsystem (`template-normalizer.js`) that pre-processes template HTML before DOM parsing (PascalCase conversion, self-closing expansion)
- **Is_Expression**: The reactive expression bound to the `:is` attribute of a `<component>` element, which evaluates to a custom element tag name string
- **Swap_Effect**: The reactive effect generated by the compiler that observes the Is_Expression and replaces the rendered element when the value changes
- **Anchor_Comment**: A DOM comment node used as a positional marker in the rendered output, allowing the Swap_Effect to know where to insert or remove the dynamic element

## Requirements

### Requirement 1: Template Syntax Recognition

**User Story:** As a developer, I want to write `<component :is="expr">` in my template, so that I can render a custom element whose tag name is determined at runtime.

#### Acceptance Criteria

1. WHEN a `<component>` element with a `:is` attribute appears in the template, THE Tree_Walker SHALL recognize it as a Dynamic_Component binding
2. THE Template_Normalizer SHALL preserve the `<component>` tag name without converting it to kebab-case or treating it as a PascalCase component import
3. WHEN a `<component>` element lacks a `:is` attribute, THE Compiler SHALL emit a compilation error indicating that `:is` is required on `<component>` elements
4. THE Tree_Walker SHALL extract the Is_Expression string from the `:is` attribute value

### Requirement 2: Reactive Element Swapping

**User Story:** As a developer, I want the rendered element to update automatically when the `:is` expression changes, so that my UI reflects the current state without manual DOM manipulation.

#### Acceptance Criteria

1. WHEN the Is_Expression evaluates to a non-empty tag name string, THE Swap_Effect SHALL create a new element using `document.createElement` with that tag name and insert it into the DOM at the Anchor_Comment position
2. WHEN the Is_Expression value changes from one tag name to another, THE Swap_Effect SHALL remove the previously rendered element from the DOM and create a new element with the new tag name
3. WHEN the Is_Expression value changes to an empty string or a falsy value, THE Swap_Effect SHALL remove the currently rendered element from the DOM without inserting a replacement
4. WHEN the Is_Expression value changes from a falsy value to a non-empty tag name, THE Swap_Effect SHALL create and insert a new element at the Anchor_Comment position
5. THE Swap_Effect SHALL execute within a reactive effect so that changes to signals, computeds, or props referenced in the Is_Expression trigger a re-evaluation

### Requirement 3: Lifecycle Callback Integrity

**User Story:** As a developer, I want `connectedCallback` and `disconnectedCallback` to fire correctly on dynamic components, so that my components can manage resources during their lifecycle.

#### Acceptance Criteria

1. WHEN a new element is created and inserted by the Swap_Effect, THE element's `connectedCallback` SHALL fire as part of the DOM insertion
2. WHEN an existing element is removed by the Swap_Effect, THE element's `disconnectedCallback` SHALL fire as part of the DOM removal
3. WHEN the Is_Expression changes from one tag to another, THE old element's `disconnectedCallback` SHALL fire before the new element's `connectedCallback` fires

### Requirement 4: Prop Forwarding

**User Story:** As a developer, I want to pass props to the dynamic component using `:attr="expr"` syntax, so that I can configure the rendered component with reactive data.

#### Acceptance Criteria

1. WHEN a `<component :is="expr">` element has additional `:attr="expr"` bindings, THE Swap_Effect SHALL set those attributes on the newly created element after creation
2. WHEN the Is_Expression changes and a new element is created, THE Swap_Effect SHALL apply all prop bindings to the new element
3. WHEN a prop expression changes while the dynamic element is rendered, THE Compiler SHALL update the attribute on the current element reactively
4. THE Compiler SHALL support multiple prop bindings on a single `<component>` element

### Requirement 5: Event Binding

**User Story:** As a developer, I want to attach event listeners to the dynamic component using `@event="handler"` syntax, so that I can respond to events from whichever component is currently rendered.

#### Acceptance Criteria

1. WHEN a `<component :is="expr">` element has `@event="handler"` bindings, THE Swap_Effect SHALL attach the event listeners to the newly created element after creation
2. WHEN the Is_Expression changes and the old element is removed, THE Swap_Effect SHALL remove event listeners from the old element before removal
3. WHEN the Is_Expression changes and a new element is created, THE Swap_Effect SHALL attach event listeners to the new element
4. THE Compiler SHALL support multiple event bindings on a single `<component>` element

### Requirement 6: Expression Source Compatibility

**User Story:** As a developer, I want the `:is` expression to work with signals, computeds, and props, so that I can use any reactive source to drive dynamic rendering.

#### Acceptance Criteria

1. WHEN the Is_Expression references a signal (e.g., `:is="currentTag()"`), THE Swap_Effect SHALL re-evaluate when the signal value changes
2. WHEN the Is_Expression references a computed (e.g., `:is="routeComponent()"`), THE Swap_Effect SHALL re-evaluate when the computed value changes
3. WHEN the Is_Expression references a prop (e.g., `:is="props.view"`), THE Swap_Effect SHALL re-evaluate when the prop value changes
4. WHEN the Is_Expression is an inline expression (e.g., `:is="isAdmin() ? 'admin-panel' : 'user-panel'"`), THE Swap_Effect SHALL re-evaluate when any referenced reactive source changes

### Requirement 7: Composition with Directives

**User Story:** As a developer, I want `<component :is="expr">` to work inside `if`/`else` blocks and `each` loops, so that I can combine dynamic rendering with conditional and iterative logic.

#### Acceptance Criteria

1. WHEN a `<component :is="expr">` element appears inside an `if` conditional branch, THE Compiler SHALL generate a valid Swap_Effect that activates when the branch is rendered
2. WHEN a `<component :is="expr">` element appears inside an `each` loop, THE Compiler SHALL generate an independent Swap_Effect for each loop iteration
3. WHEN the containing `if` branch becomes inactive, THE Swap_Effect SHALL clean up the dynamic element (triggering `disconnectedCallback`)
4. WHEN the containing `each` loop removes an item, THE Swap_Effect for that iteration SHALL clean up the dynamic element

### Requirement 8: Bundle Mode Compatibility

**User Story:** As a developer, I want `<component :is="expr">` to work in `--bundle` mode (IIFE output, `file://` protocol), so that dynamic components function in all deployment scenarios.

#### Acceptance Criteria

1. WHEN the Compiler runs in bundle mode (`--bundle`), THE compiled output containing Dynamic_Component bindings SHALL produce valid IIFE code
2. WHEN the compiled bundle is loaded via `file://` protocol, THE Swap_Effect SHALL create elements using `document.createElement` without requiring module resolution
3. THE Compiler SHALL ensure all components referenced by the Is_Expression at runtime are registered in the custom elements registry before the Swap_Effect executes (via import order or bundle structure)

### Requirement 9: Compile-Time Behavior

**User Story:** As a developer, I want the compiler to accept any valid expression in `:is` without requiring static resolution, so that I can use fully dynamic tag names determined at runtime.

#### Acceptance Criteria

1. THE Compiler SHALL NOT attempt to statically resolve the value of the Is_Expression at compile time
2. THE Compiler SHALL NOT emit an error when the Is_Expression references variables, function calls, or complex expressions that cannot be evaluated statically
3. WHEN the Is_Expression evaluates to a tag name that is not registered in the custom elements registry at runtime, THE Swap_Effect SHALL create an `HTMLUnknownElement` (standard browser behavior for `document.createElement` with an unregistered tag)
4. THE Compiler SHALL generate code that defers all tag name validation to runtime

### Requirement 10: Codegen Output Structure

**User Story:** As a developer, I want the compiled output for dynamic components to follow the same patterns as other directives (if, each, show), so that the generated code is consistent and maintainable.

#### Acceptance Criteria

1. THE Codegen SHALL emit an Anchor_Comment node in the template string to mark the position of the Dynamic_Component
2. THE Codegen SHALL emit a reactive effect that reads the Is_Expression, compares it to the previous tag name, and performs the DOM swap when the value changes
3. THE Codegen SHALL store a reference to the currently rendered element so it can be removed on the next swap
4. WHEN the component has prop bindings, THE Codegen SHALL emit attribute-setting code inside the swap logic that runs after element creation
5. WHEN the component has event bindings, THE Codegen SHALL emit `addEventListener` calls inside the swap logic that run after element creation
